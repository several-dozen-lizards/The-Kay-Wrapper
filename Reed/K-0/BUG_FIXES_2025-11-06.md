## Fix 1: Threading Error in kay_ui.py Import Window

### Problem

```
Exception in thread Thread-1 (run_async_import):
_tkinter.TclError: invalid command name ".!importwindow.!ctkframe4.!ctkbutton"
```

**Root Cause:** The import process runs in a background thread via `asyncio.run()`, but UI widgets were being updated directly from that thread. Tkinter **REQUIRES** all UI updates happen on the main thread.

### Location

**File:** `kay_ui.py`
- Line 248-257: `log()` method
- Lines 299-302: Button updates in `run_import()` finally block

### Code Changes

#### Changed 1: Made `log()` method thread-safe

**BEFORE (line 248-253):**
```python
def log(self, message):
    """Add message to progress log."""
    self.progress_text.configure(state="normal")
    self.progress_text.insert("end", f"{message}\n")
    self.progress_text.see("end")
    self.progress_text.configure(state="disabled")
```

**AFTER (line 248-257):**
```python
def log(self, message):
    """Add message to progress log (thread-safe)."""
    def _update_ui():
        self.progress_text.configure(state="normal")
        self.progress_text.insert("end", f"{message}\n")
        self.progress_text.see("end")
        self.progress_text.configure(state="disabled")

    # Schedule UI update on main thread
    self.after(0, _update_ui)
```

#### Changed 2: Made button updates thread-safe

**BEFORE (lines 296-297):**
```python
finally:
    self.start_button.configure(state="normal")
    self.cancel_button.configure(state="disabled")
```

**AFTER (lines 300-302):**
```python
finally:
    # Schedule button updates on main thread
    self.after(0, lambda: self.start_button.configure(state="normal"))
    self.after(0, lambda: self.cancel_button.configure(state="disabled"))
```

### Technical Explanation

**Why `self.after(0, callback)` works:**

Tkinter's `.after(milliseconds, callback)` schedules a function to run on the main thread's event loop. Using `after(0, ...)` schedules it immediately (0ms delay) but ensures it runs on the main thread.

This pattern is the standard way to make thread-safe UI updates in Tkinter:
1. Background thread does work
2. Instead of updating UI directly, schedules update via `.after()`
3. Main thread's event loop picks up the scheduled update
4. UI widgets update safely on main thread

**Why direct updates fail:**

Tkinter widgets are not thread-safe. When you call `.configure()`, `.insert()`, etc. from a background thread, Tkinter's internal state becomes corrupted, leading to:
- `TclError: invalid command name` (widget was destroyed/invalid)
- Segmentation faults
- UI freezes
- Unpredictable behavior

### Result

✓ Import window no longer crashes during file import
✓ Progress log updates safely from background thread
✓ Button states update safely after import completion
✓ No more `TclError` exceptions

## Fix 2: Response Length Configuration

### Problem

Kay's responses were being cut off mid-sentence at around 400-450 characters (6 sentences), even when comprehensive answers were appropriate.

### Root Cause Discovery

**Investigation revealed the actual culprit:**
1. `max_tokens` was already set to **8192** (maximum supported) - NOT the issue
2. **ACTUAL ISSUE:** `_limit_length_by_affect()` function in `kay_ui.py` was artificially truncating responses!
   - Line 587-591: Calculated max sentences based on affect slider
   - With affect=3.5: max_sents = max(2, int(2 + 3.5 * 1.4)) = 6 sentences
   - Each sentence ~70-75 chars = ~420-450 chars total
   - This matched the observed truncation length exactly!

### Analysis

The function was designed to scale response length with emotional affect, but it was causing premature truncation:
- Affect 1.0 → 3 sentences max (~210 chars)
- Affect 2.0 → 4 sentences max (~280 chars)
- Affect 3.5 → 6 sentences max (~420 chars) ← User's experience
- Affect 5.0 → 9 sentences max (~630 chars)

Even at maximum affect (5.0), responses were capped at 9 sentences, preventing comprehensive answers.

### Location

**Primary Fix:**
- **File:** `kay_ui.py`
- **Line 587-593:** `_limit_length_by_affect()` function - Disabled artificial truncation

**Secondary Enhancement:**
- **File:** `integrations/llm_integration.py`
- **Lines 108-120:** `_style_block()` function - Enhanced response length instructions
- **Line 456:** Added logging for max_tokens visibility

### Code Changes

#### Changed 1: DISABLED artificial response truncation in kay_ui.py

## Technical Notes

### Response Length Truncation Issue

**The Problem:**
The `_limit_length_by_affect()` function was designed to scale response length with emotional affect, but it was causing severe truncation:

```python
# OLD CODE (BROKEN):
def _limit_length_by_affect(self, text: str) -> str:
    a = float(self.affect_var.get())
    max_sents = max(2, int(2 + a * 1.4))  # 2..9 sentences max
    parts = re.split(r'(?<=[\.\?!])\s+', text.strip())
    return " ".join(parts[:max_sents])  # Artificially truncate!
```

**Impact by affect level:**
- Affect 1.0 → 3 sentences max (~210 characters)
- Affect 2.0 → 4 sentences max (~280 characters)
- Affect 3.0 → 5 sentences max (~350 characters)
- Affect 3.5 → 6 sentences max (~420 characters) ← Default setting
- Affect 4.0 → 7 sentences max (~490 characters)
- Affect 5.0 → 9 sentences max (~630 characters)

Even at maximum affect (5.0), responses were capped at just 9 sentences (~630 chars), preventing Kay from giving comprehensive answers about complex topics.

**Why this was bad:**
1. User asks: "Tell me about the pigeons"
2. LLM generates 15 sentences with rich detail
3. `_limit_length_by_affect()` truncates to 6 sentences
4. Result: Cut-off mid-thought, missing important context

**The Solution:**
Disabled the function entirely. Response length is now controlled by:
1. LLM prompt instructions (3-5 sentence minimum baseline)
2. `max_tokens=8192` (allows ~6000 words)
3. LLM's natural response pacing based on topic complexity

### Tkinter Threading Best Practices

**DO:**
- Use `self.after(0, callback)` to schedule UI updates from background threads
- Keep UI code on main thread
- Use threading for long-running operations (file I/O, network, heavy computation)

**DON'T:**
- Call widget methods (`.configure()`, `.insert()`, etc.) directly from background threads
- Access Tkinter variables from background threads without synchronization
- Create Tkinter widgets from background threads
